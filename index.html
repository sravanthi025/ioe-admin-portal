IOE Admin Portal — User Guide
NxtWave · Intensive Offline Exam Operations
What is this app?
NxtWave Admin — Intensive Offline Exam (IOE) Admin Portal

This is an internal operations tool for managing the complete lifecycle of Intensive Offline Exam (IOE) project assignments. It coordinates five teams — On Ground, Content, Assessment Ops, Instructors, and Admin — through two structured workflows: the Assignment Evaluation workflow (request → links → auto-allocation → evaluation → export) and the SLA Pipeline (syllabus → config → Topin go-live tracking).

Access is role-gated — an Admin must approve every new user and assign them a team role before they can use the portal. Each role sees only the pages relevant to their responsibilities.

ES modules via CDN, no build step

How it works
Two parallel workflows managed through this portal.
Workflow A — Assignment Evaluation
1
Raise Assignment Request On Ground Team

Specifies Phase, Batch, Week, number of sets, subjects/topics, and whether a Mock assignment is needed. Notifies Content Team.

2
Submit Assignment Links Content Team

Uploads one assignment question document link per set (Set 1, Set 2, …). Submitting triggers automatic student set allocation.

3
Auto Set Allocation System (automatic)

Students in that Phase+Batch are sorted by UID and distributed round-robin across sets. On Ground Team can immediately download the allocation CSV.

4
Import Student Submissions Assessment Ops Team

Students submit GitHub and Deployment links via Google Form. Ops imports the CSV — links matched by Candidate UID are saved to each student's record.

5
Evaluate Students Instructor

Instructor opens each student's scoring form — 10 criteria across 4 sections on a 0–5 scale. Part averages, total (/20), and performance band are auto-calculated. Predefined remark chips speed up feedback.

6
Export Evaluation CSV Instructor · Admin

Download the complete evaluation CSV — 26 columns matching the Google Sheet format including all criterion scores, averages, total, performance band, and remarks.

Workflow B — Topin Assessment Publishing & Evaluation
1
Submit Syllabus On Ground Team

Phase, Batch, Week, assessment date, start/end time, subjects, topics, and Mock required flag. Starts the SLA clock. Go to Syllabus page.

2
Submit Mock Config Link Content Team

Topin config URL for the Mock assessment. Go to Topin Configs page. Skipped if Mock not required.

3
Publish Mock & Mark Live Assessment Ops Team

Publish the Mock assessment on Topin, then click Mark Mock Live on the Assessments page. Timestamps the SLA step. Skipped if Mock not required.

4
Students Take Mock Assessment Students (on assessment date)

Students complete the Mock on Topin. On Ground Team monitors attendance. Results are available on Topin for review. Skipped if Mock not required.

5
Submit Main Config Link Content Team

Topin config URL for the Main assessment. Go to Topin Configs page. Submitted after the Mock assessment is complete.

6
Publish Main & Mark Live Assessment Ops Team

Publish the Main assessment on Topin, then click Mark Main Live. Missed deadline turns the step red on Assessment Details and triggers a breach notification.

7
Students Take Main Assessment Students (on assessment date)

Students complete the Main assessment on Topin within the scheduled window (start/end time from Syllabus). On Ground Team monitors.

8
Review Results & Acknowledge Breaches Admin · Assessment Ops Team

Assessment results are available on Topin for review. On Assessment Details, acknowledge any SLA breaches with "Mark as Read" to clear the red alert. Full pipeline status (green / amber / red per step) is preserved for reporting.

Steps 2, 3, and 4 (Mock) are skipped when "Mock Required" is unchecked in the Syllabus. Without Mock the pipeline is: Syllabus → Main Config → Main Live → Students Take Main → Review Results.

Assessment Details All roles
assessment-details

The SLA control room. Shows the full 5-step pipeline status for every assessment — who did what and whether they hit the deadline.

Pipeline Steps

Student Data On Ground Team · Admin
students

The student roster for each Phase, Batch, and Week. This is the source of truth for set allocation — when assignment links are submitted, the portal pulls students from here to distribute across sets.

Add Students

Upload via Excel file or add individually. Required fields: Name, UID, Student ID, Email, Phase, Batch. P3 and P4 batches also need a Domain (Python / Java).

Search & Filter

Filter the table by Phase, Batch, and Week. Student count shown in the Dashboard stat card. Used as the source for round-robin set allocation when assignment links are submitted.

Student data must be uploaded before assignment links are submitted — allocation queries this collection. If students are missing, the allocation count will be lower than expected.
Syllabus On Ground Team · Content Team · Admin
syllabus

The starting point of the SLA pipeline. On Ground Team submits the weekly syllabus for each Phase/Batch/Week. Every assessment on Topin begins here — submitting a syllabus starts the SLA clock.

Add Syllabus

Fill in Phase, Batch, Week, subjects and topics, assessment date, start/end time, and whether a Mock assessment is required. The Mock flag controls whether the 5-step or 3-step SLA pipeline is shown.

SLA Clock Starts Here

The moment a syllabus is submitted, the SLA timer begins. Content Team is expected to submit config links within the configured SLA window. Editing an existing syllabus updates the downstream records.

Important: If "Mock Required" is checked, Assessment Details will show 5 steps (Syllabus → Mock Config → Mock Live → Main Config → Main Live). If unchecked, only 3 steps are shown (Syllabus → Main Config → Main Live).
Topin Configs Content Team · Admin
configs

Content Team submits the Topin assessment configuration URLs after reviewing the syllabus. These links are used by Assessment Ops to publish the assessments on the Topin platform.

Mock & Main Config Links

Each syllabus entry gets two config links — one for Mock and one for Main. Content Team submits them separately. Each submission records a timestamp for SLA tracking.

Linked to Syllabus

Each config entry is linked to a specific Phase/Batch/Week syllabus. Submitting a config link advances the SLA pipeline from Step 1 to Step 2 (Mock) or Step 4 (Main) in Assessment Details.

Config links must be submitted before Assessment Ops can mark the assessment as Live. If a config link is missing for a syllabus, Assessment Details will show that step as pending or breached.
Assessments Assessment Ops Team · Admin
assessments

Assessment Ops marks each Topin assessment as Live and tracks the complete publish-and-evaluate lifecycle. Mock and Main go-live events are tracked separately with timestamps for SLA compliance.

Mark Mock / Main Live

Click Mark Mock Live or Mark Main Live once published on Topin. Timestamp recorded immediately — SLA step on Assessment Details turns green (on time) or amber (late).

Pending Assessments View

All assessments awaiting go-live are listed here, sorted by date. Missed deadlines appear as red breach rows and trigger notifications to all roles.

Post-Live: Students Take Assessment

Once marked Live, students complete the assessment on Topin within the scheduled window (date + start/end time from Syllabus). On Ground Team monitors attendance on the day.

Review Results on Topin

After students complete the assessment, results and scores are available directly on the Topin platform. Go to Assessment Details in this portal to acknowledge any SLA breaches with "Mark as Read" to close the alert.

Always mark Live here immediately after publishing on Topin — the SLA breach timer runs from the moment the config link was submitted. Delayed marking inflates breach counts on Assessment Details.

Pending Approvals

Evaluation Scoring System
How the 10-criterion form works and how totals are calculated.

Each criterion is scored 0 – 5 using the score picker buttons. Part averages and the total update live. Maximum total is 20 + up to 5 bonus.

Part 1 — Design & UI Implementation (avg /5)
  • Layout & visual consistency
  • Component styling & consistency
  • Responsiveness (all screen sizes)
Part 2 — Functional Correctness (avg /5)
  • Core feature completion
  • Interactive behavior (navigation, forms, controls, scroll, etc.)
  • Dynamic content handling (search/filter/pagination where applicable)
Part 3 — Data Handling & State Management (score /5)
  • Data Handling & State Management
Part 4 — Code Quality & Delivery (avg /5)
  • Component structure & maintainability
  • Readability & best practices
  • Submission quality (working build, repo, README, no errors)
Total = Part1 Avg + Part2 Avg + Part3 Score + Part4 Avg + Bonus — Performance Bands:

Key Concepts
Terms and logic you'll encounter throughout the portal.
Phase / Batch / Week

Three-level grouping for students. Phase is the program level (P1–P4). Batch is the cohort within a phase. Week is the assignment cycle. Together they identify which students receive a specific assignment.

Set Allocation (Round-Robin)

Students sorted by UID are distributed cyclically across sets: student 1 → Set 1, student 2 → Set 2, … student N+1 → Set 1 again. Triggered automatically the moment Content Team submits links. Ensures even distribution.

SLA Pipeline

Five-step (or three-step without Mock) workflow for Topin assessment publishing. Each step has a configured deadline. Missing a deadline triggers a red breach notification visible to all roles on Assessment Details.

Performance Band

Auto-calculated from total score: ≥17 Exceptional (Strong Hire) · ≥14 Strong (Hire) · ≥11 Competent (Hire with Reservations) · ≥8 Borderline (Conditional) · <8 Significantly Below (Do Not Hire).

Notifications

Bell icon in the top bar shows unread count. Triggers: assignment request raised → notifies Content Team + Admin. Links submitted → notifies On Ground Team + Ops + Instructor + Admin. Notification panel shows timestamp and context.

Evaluation CSV Export

26-column format matching the Google Sheet exactly: 8 Assignment Detail columns, 4×Part1, 4×Part2, 2×Part3, 4×Part4, Bonus, Total, Performance Band, Remarks. Scores exported as "N : Label" (e.g., "4 : Strong implementation").

Role Guide
On Ground Team
Syllabus submission · Assignment requests · Set allocation download

Pages you can access

Dashboard · Student Data · Syllabus · Assignments · Assessment Details · About

SLA workflow — your steps

  1. Go to Student Data and upload the student roster for your Phase/Batch before the assessment week
  2. Go to Syllabus and click Add Syllabus — fill Phase, Batch, Week, subjects/topics, assessment date, start/end time, and check "Mock Required" if applicable. This starts the SLA clock
  3. Check Assessment Details to track SLA status — green steps are on time, red means a breach

Assignment workflow — your steps

  1. Go to Assignments and click Raise Request — fill Phase, Batch, Week, number of sets, subjects/topics, and Mock flag
  2. Content Team will be notified. Status shows "Pending Links" until they respond
  3. Once Content Team submits links, status updates to "X students allocated" — students have been auto-assigned to sets
  4. Click Download CSV to get the allocation sheet: Student Name, UID, Student ID, Email, Allocated Set, Assignment Link
  5. Share assignment links with students via WebEngage or directly
Tip: Student data must be uploaded before assignment links are submitted — the system pulls students from that page for set allocation.
Role Guide
Content Team
Config link submission · Assignment question link submission

Pages you can access

Dashboard · Syllabus (view) · Topin Configs · Assignments · Assessment Details · About

SLA workflow — your steps

  1. Once On Ground Team submits a syllabus, go to Topin Configs
  2. Find the Phase/Batch/Week entry and click Add Config
  3. Submit the Mock Config URL first (if Mock Required), then the Main Config URL — each submission timestamps the SLA pipeline step
  4. Check Assessment Details to confirm the steps turned green

Assignment workflow — your steps

  1. You receive a notification when On Ground Team raises an assignment request
  2. Go to Assignments — rows with status "Pending Links" need your action
  3. Click Submit Links — enter a label and URL for each set (e.g., Set 1 → Google Doc link)
  4. Click Save Links — this triggers student set allocation automatically and notifies On Ground Team, Assessment Ops, and Instructors
Important: Submit config links as soon as possible after the syllabus is received — the SLA breach timer is running from the moment the syllabus was submitted.
Role Guide
Assessment Ops Team
Mark assessments live · Import student submissions

Pages you can access

Dashboard · Assessments · Assignments · Assessment Details · About

SLA workflow — your steps

  1. Once Content Team submits config links, go to Assessments
  2. Find the row for the relevant Phase/Batch/Week
  3. Publish the assessment on Topin, then come back and click Mark Mock Live (if Mock Required) and/or Mark Main Live
  4. The SLA pipeline on Assessment Details will update to green — or amber if you're submitting after the deadline

Assignment workflow — your steps

  1. Once students submit GitHub and Deployment links via Google Form, download the form responses as a CSV
  2. Go to Assignments and click Evaluate (X/Y) on the relevant row
  3. In the student list modal, click Import Submissions CSV
  4. Select the downloaded CSV — links matched by Candidate UID are saved automatically
CSV columns the importer looks for: "Candidate UID" (or "uid"), "Deployment Link" (or "deploy"), "GitHub Link" (or "git"). Case-insensitive, partial matches work.
Role Guide
Instructor
Student evaluation · CSV export

Pages you can access

Dashboard · Assignments · Assessment Details · About

Your step-by-step workflow

  1. You receive a notification when assignment links are submitted and students are allocated
  2. Go to Assignments — rows show Evaluate (X/Y) where X = evaluated so far, Y = total allocated
  3. Click Evaluate to open the student list — each row shows Name, UID, Set, submission links (Deploy ↗ / Git ↗), and current score if already evaluated
  4. Click Evaluate on any student to open the scoring form. The Assignment Details card at the top shows Batch, Week, Set, and the Question Link for reference
  5. Verify the student's Deployment and GitHub links are pre-filled (imported by Ops). Update them if missing
  6. Score each of the 10 criteria using the 0–5 buttons — part averages, total (/20), and performance band update in real time
  7. Click remark chips to quickly add common observations, or type custom remarks in the text box
  8. Click Save Evaluation — the score and band appear in the student list immediately
  9. Repeat for all students. Click Export Eval CSV to download the complete evaluation in Google Sheet format (26 columns)
All 10 criteria must be scored before saving. Score guide: 0 = Broken · 1 = Very weak · 2 = Partial · 3 = Minimum · 4 = Strong · 5 = Excellent. The score legend is also shown at the top of the scoring form.
